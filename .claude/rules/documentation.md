# Documentation

Rules for the user-facing documentation site under `documentation/`. For prose and terminology, follow `writing.md`; for Markdown formatting, follow `markdown.md`.

## Source vs. Generated

- Edit only the sources under `documentation/docs/`.
- `documentation/site/`, `documentation/.cache/`, and `documentation/docs/version.json` are generated output; never edit them by hand.

## Examples and Snippets

- Public-facing example domains live under `documentation/guides/<slug>/model/` and are lint-gated; that is the home for teachable example models.
- When showing snippets from a public example, embed them via `pymdownx.snippets` (`--8<-- "<slug>/model/<file>.esdm.yaml"`) rather than copying the YAML inline. The snippet stays in sync with the linted source.
- The Reference pages embed per-kind schema excerpts generated by `make generate-reference-snippets`; regenerate them instead of editing the excerpts by hand.

## Page Style

These apply to guides and reference pages.

- Documentation is instructional: lead with the high-level concept, then add detail and show how to use it.
- Pages of the same kind (for example the getting-started guides) follow the same structure; predictable layouts make pages easier to navigate.
- Lists are welcome for steps, options, and enumerations.
- Where it helps the reader, end a page with a "For more information" pointer to related pages.

## Admonitions

The docs site is mkdocs-material. Use admonitions sparingly. The documentation uses these types, each either as an open `!!!` block or a collapsed `??? "Title"` block:

- `info` and `note` for factual side notes.
- `warning` for pitfalls and surprises; `danger` for stronger cautions, such as an API or operation that must not be misused.
- `tip` for best-practice hints.
- `question` for rhetorical or FAQ-style entries.
- `quote` for quoted or cited material.
- `success`, typically collapsed, for grouped or optional content the reader can expand on demand.
- Do not reach for the remaining Material types (`abstract`, `example`, `bug`, `failure`) without a clear reason.

## Diagrams

- Use a diagram only when it earns its keep didactically.
- Keep diagrams as inline SVG or as repository-hosted images under `documentation/docs/images/`; never embed an external image.
