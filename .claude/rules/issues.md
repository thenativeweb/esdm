# Issues

## Creating Issues

- Create every issue from one of the repository's templates and keep its structure: pick `bug`, `feature`, or `task` to match the kind of work.
- Set the GitHub issue type to match the template, and keep the two in sync: Bug <-> bug, Feature <-> feature, Task <-> task. The type and the template must never disagree.
- Pick the kind by intent:
  - **Bug**: erroneous or unexpected behavior that needs to be fixed.
  - **Feature**: new functionality to be added.
  - **Task**: a specific piece of work that is neither, for example a dependency update, documentation, a refactoring, or a concept/discussion issue.

## Sections

- The first two sections are mandatory:
  - **"What is this ... about?"** describes the issue.
  - **"What needs to be done ...?"** lists concrete, verifiable steps as a checklist.
- **"What else should we know?"** is optional. Use it when there is something worth keeping next to the issue, for example:
  - links to further or background information that is helpful as context,
  - background knowledge that is useful but not required to do the work,
  - the customer or contact an issue originated from, so we can follow up or ask back if in doubt.
